# Java Spring 代码编写技能
概述
使用本技能编写 Java/Spring 代码，需要优先适配当前代码仓库的已有规范，再遵循通用稳定的工程编码准则。  
在研读本地代码与仓库规范前，禁止主观预设全局通用父类、统一返回封装类、DTO 命名后缀、DAO 层目录结构。  
如果输入内容来自其他技能产出的「当前步骤 / 任务实现概要」，需要将这份概要作为本轮编码的约束契约，仅完成该工作范围内的开发任务。  
工作流程  
## 1. 编码前先梳理项目上下文  
优先读取仓库规范文件。  
若项目中存在以下文件，务必依次查阅：AGENTS.md、.claude/rules/**/*.md、CLAUDE.md、.claude/CLAUDE.md、CLAUDE.local.md。  
由于仓库中存量代码规范较差，不可盲目采用，需要遵循本规范。  
在确定以下编码规则前，需要先查看当前模块内的邻近代码文件：  
接口统一返回封装类类型  
控制器基础父类  
DTO/VO 的命名规则与存放包路径  
实体类的继承方案  
选用原生ServiceImpl还是项目自定义BaseServiceImpl  
映射文件、基础设施层、仓储层、XML 文件的目录结构  
异常工具类与错误码编码规范  
日志打印格式规范  
是否需要保留原有旧代码，最小化代码提交变更量  
## 2. 区分通用稳定规则与项目个性化规则  
以下规则默认作为通用稳定规范统一执行：  
新增代码统一使用构造函数注入  
遵循瘦控制器设计，业务逻辑下沉至 Service 层  
请求入参必须做参数校验  
日志采用占位符格式化写法  
业务场景优先使用项目自定义业务异常，禁止直接抛出原生RuntimeException  
对象转换优先使用 MapStruct，避免重复手动赋值  
采用卫语句、精简方法粒度、简化分支逻辑  
数据写入类接口必须考虑幂等性与并发安全  
若项目已使用 JSON 类型字段，需要做查询优化并使用强类型 JSON 字段处理方案  
以下属于项目个性化规则，必须先从现有代码中调研确认，不能主观默认：  
统一返回封装类：FwkApiResponse或其他自定义返回类  
控制器基础父类：BaseMamaController、FwkBaseController或其他自定义父类  
入参出参命名规范：Req/Res、Req/Rsp、DTO、VO或混合命名方式  
v2/dao目录结构、项目是否划分基础设施层 / 仓储层  
实体类需要继承哪一个基础父类  
审计字段填充工具类（例如OperatorUtil）  
数据表代码生成时，是否需要同步生成 SDK、JPA 相关类  
禁止用全局通用规范覆盖项目已落地的本地编码规范。  
## 3. 选择对应的编码实现策略  
现有模块内修改代码：参考同模块内维护完善的邻近文件编码风格，除非该代码存在明显缺陷、和仓库强制规范冲突  
成熟模块内新建文件：遵循当前模块已有的命名方式、包路径、类继承规则、统一返回封装类规范  
全新业务模块无历史代码参考：使用基础通用编码规范，编写简洁现代化的 Spring 代码，减少冗余模板代码  
任务来自任务实现概要文档：严格遵守文档划定的非业务范围边界，禁止擅自扩展后续任务、关联迭代需求  
## 4. 编码规则落地要求  
控制器规范  
所有业务逻辑禁止写在 Controller 中  
需要参数校验的接口添加@Validated注解  
优先复用项目内已有的接口成功响应静态构建方法  
明确定义接口请求、响应参数类型  
DTO/VO 对象规范  
严格遵循项目的类后缀规则与包存放路径  
若存在嵌套 DTO 类且可能和 Swagger/OpenAPI 文档出现命名冲突，需要给嵌套的Req/Res类增加业务前缀  
仅当项目已有大量使用案例、或可以优化构建类对象映射场景时，才使用@Accessors(chain = true)链式注解  
业务服务层规范  
代码使用卫语句优化分支  
抽取重复分支逻辑为公共方法  
批量数据操作优先使用批量接口，禁止循环单条写入  
数据变更类接口必须显式处理幂等、并发问题  
持久层规范  
遵循当前模块已有的Entity实体、InfraService基础设施服务继承体系  
数据库 JSON 类型字段，项目已有成熟方案时，使用强类型字段搭配JacksonTypeHandler、autoResultMap = true配置  
仅当模块内已在使用时，复用项目封装的BaseServiceImpl基础服务类  
对象映射规范  
DTO 与实体类之间的转换优先使用 MapStruct  
在映射接口中定义集合转换方法，禁止手写循环做集合对象转换  
错误码及异常处理需要定义准确的错误码值code及描述
## 5. 数据表驱动代码生成规范  
根据数据表生成代码前，必须先调研目标模块规范，不能所有项目套用同一套代码模板，需要确认模块是否需要生成以下内容：  
仅 JPA 实体类  
仅 MyBatis Plus 实体类  
同时生成 SDK 依赖类与运行时业务类  
Mapper XML 映射文件  
自定义基础实体类、基础服务类  
仓储层包装类  
数据表生成文档仅作为初始模板参考，不能作为全仓库通用强制标准。

查阅references/java-spring-runtime-topics.md：学习并发、缓存、配置安全、异步执行、RocketMQ 事务、权限、性能相关规范

编码收尾快速自检项  
代码编写完成前必须校验以下内容：  
代码完全符合仓库明文强制规范  
代码风格和当前模块邻近已有代码保持统一  
没有在无代码依据的前提下硬编码项目个性化规则  
请求参数校验、日志打印、异常处理流程逻辑统一通顺  
新增类、类后缀、包路径符合当前模块编码规范
