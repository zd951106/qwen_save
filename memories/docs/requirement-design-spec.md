# 需求设计与系统设计规范

> **文档说明**
> 本规范基于实际项目经验提炼而成，已去除具体业务名词，适用于各类项目和需求的系统设计与开发，不限于特定任务类型或行业领域。
>
> **用途**：跨项目通用规范。向开发人员下发开发任务时，按此规范提供需求信息，确保设计决策有充分输入，减少返工和过度工程。
> **使用方式**：不必每次全部覆盖，按任务复杂度取用相关章节。简单任务填第一+第九部分即可；复杂任务建议覆盖全部。对于核心资金链路，应覆盖全部章节（含补充章节）。
> **最后更新**：2026-07-09

## 第一部分：需求设计

### 1. 业务背景与目标

**规范要求**：
- 用1-2句话描述：解决什么业务问题、最终交付物是什么
- 明确上下游：谁触发、谁消费、格式由谁定义
- 说明是否为一次性交付还是长期维护

**为什么重要**：没有背景，设计者容易按"通用最佳实践"设计，可能与实际诉求偏离。例如目标是"对端拉取数据合并"时，应设计成文件交换而非推送 API；目标是"前端实时查询"时，应设计成 REST 接口而非异步任务。

**提供方式**：
```
背景：[业务场景]
目标：[交付物形态]
上下游：[触发方 → 处理方 → 消费方]
```

### 2. 对接系统协议规范

**规范要求**：
- 如果对接外部系统，提供完整的格式规范：样例数据/样例文件、接口文档、对端代码
- 明确样例与代码不一致时的优先级
- 列出所有字段，标注哪些已确认、哪些待确认

**为什么重要**：实战中字段映射 TODO 可达十余处，根因是初期只有样例缺少对端代码。对端代码或接口定义应在需求阶段就获取，而非编码后才发现字段含义不明。

**提供方式**：
```
对端系统：[系统名]
格式规范来源：[样例数据 / 接口文档 / 对端代码 / Schema定义]
样例参考：[路径或URL]
对端代码/接口定义：[路径]
已知差异：[样例与代码不一致处，以哪个为准]
```

### 3. 功能边界与粒度

**规范要求**：
- 明确处理维度（按商户/机构/日期/渠道/租户等）
- 明确每个维度产出几个结果
- 明确边界数据如何处理（空数据/异常数据/跨边界数据）

**为什么重要**：维度和粒度直接影响输出数量、并发模型、分页策略。空数据处理方式取决于对端是否检查结果存在——跳过可能导致对端认为数据缺失。

**提供方式**：
```
处理维度：[按什么维度拆分处理]
每个维度产出：[几个结果/几条记录]
空数据处理：[生成空结果 / 跳过 / 返回默认结构]
命名规则：[如文件名、接口路径、队列名等] [N/A]
```

### 4. 触发机制与时间约束

**规范要求**：
- 明确触发方式：定时任务 / 接口调用 / 消息消费 / 手动
- 明确同步还是异步
- 明确时间窗口和超时要求
- 明确执行时段是否避开业务高峰

**为什么重要**：时间约束直接决定架构选型——1小时窗口 + 百万级数据需要并发+流式；无时间约束可串行简化。执行时段影响是否需要资源隔离。

**提供方式**：
```
触发方式：[定时任务 / REST接口 / MQ / 手动]
同步/异步：[同步等待 / 异步通知]
时间窗口：[如"凌晨1小时" / "无约束"]
超时要求：[如"55秒" / "无约束"]
执行时段：[如"凌晨低峰" / "随时"]
```

### 5. 数据量与性能基线

**规范要求**：
- 提供单次处理最大数据量、总量、日常均值
- 提供时间分布特征（高峰/低峰时段）
- 如有调研数据，提供路径

**为什么重要**：数据量是架构决策的第一输入。按最坏场景设计会导致过度工程（不必要的线程池、多层降级）；按乐观值设计会导致性能不足。应基于实际数据做工程决策。

**提供方式**：
```
单次最大：[如"98万笔" / "单次10条"]
总量：[如"27个维度合计200万笔"]
日常均值：[如"多数<20万笔"]
时间分布：[如"凌晨低峰" / "全天均匀"]
调研数据：[文件路径] [N/A]
```

## 第二部分：数据设计

### 6. 数据来源与存储架构

**规范要求**：
- 明确数据从哪张表/哪个服务获取
- 明确是否为分片表/分区表，分片键是什么
- 明确查询条件是否包含分片键（不含则广播查询）
- 明确是否有从库可读
- 提供实体类路径

**为什么重要**：分片表广播查询可能触发中间件兼容性问题（如 ResultSet 不支持按列标签取值，ORM 实体映射失败）。提前知道可直接设计原生 SQL + 索引映射，避免编码后踩坑。

**提供方式**：
```
数据表/服务：[表名或服务名]
是否分片表：[是/否，分片键，分片数] [N/A]
查询条件含分片键：[是/否]
从库：[有/无，DataSource名称] [N/A]
实体类路径：[如 xxx/entity/Order.java]
```

### 7. 字段映射与转换规则

**规范要求**：
- 逐字段列出映射关系：源字段 → 目标字段 → 转换规则
- 枚举转换需列出完整映射表
- 标注源表缺失字段及其获取方式
- 检查不同业务类型是否共用同一状态字段但数值含义不同

**为什么重要**：状态常量数值重叠是隐蔽 bug——同一数值对不同业务类型含义不同（如 status=1 对支付是"成功"对退款是"已退款"），需按业务类型分别映射。这类 bug 不会编译报错，只会在数据层面出错。

**提供方式**：
```
字段映射表：
  | 源字段 | 目标字段 | 转换规则 | 确认状态 |
  | ... | ... | ... | 已确认/TODO |

枚举映射：[如 tradeCategory=POS_CONSUME → 目标311]
缺失字段：[哪些目标字段源表没有，如何获取]
常量类冲突：[不同业务类型是否共用状态字段但含义不同]
```

### 8. 数据筛选与终态定义

**规范要求**：
- 明确哪些状态的数据纳入处理，哪些排除
- 列出所有纳入状态的码值和含义
- 提供状态常量类的路径
- 区分不同业务类型（如支付/退款/充值/提现）的终态列表

**为什么重要**：终态定义错误会导致数据遗漏或冗余。只查"成功"状态会遗漏退款/失败等终态；不区分业务类型查同一组状态会查到错误数据。

**提供方式**：
```
纳入状态：[码值列表 + 含义]
排除状态：[码值列表 + 含义]
常量类路径：[如 OrderStatusConstant / RefundOrderStatusConstant]
按业务类型区分：[是/否，各类型的终态列表] [N/A]
```

### 9. 分页与流式处理策略

**规范要求**：
- 数据量超万行时必须指定分页策略
- 游标分页（推荐）需指定游标字段、方向、初始值
- offset 分页仅适用于小数据量
- 明确 pageSize
- 流式处理需明确每页处理后的资源释放策略

**为什么重要**：深分页（OFFSET 值很大）性能急剧下降。游标分页每次查询走索引，性能稳定。流式处理 + 每页资源释放可控制内存不随数据量线性增长。

**提供方式**：
```
预估单次数据量：[如"98万行" / "百条以内"]
分页策略：[游标分页 / offset分页 / 不需要] [N/A]
游标字段：[如 order_id] [N/A]
游标方向：[正序/倒序，初始值] [N/A]
pageSize：[如10000] [N/A]
流式处理：[是/否，缓冲区大小，每页资源释放方式] [N/A]
```

### 10. 数据一致性与幂等性

**规范要求**：
- 明确重复执行是否安全（幂等性）
- 明确是否需要事务边界（多表写入时）
- 明确并发写入同一资源时的冲突处理
- 明确数据快照一致性要求（查询期间数据变化是否可接受）

**为什么重要**：任务可能因重试/补偿而重复执行，如果非幂等会产生重复结果/重复记录。多表写入不在同一事务会导致部分成功部分失败的数据不一致。

**提供方式**：
```
幂等性要求：[是/否，重复执行时的预期行为]
事务边界：[单表 / 多表同一事务 / 不需要]
并发冲突：[是否多线程写同一资源，如何避免]
快照一致性：[查询期间数据变化是否可接受]
```

### 补充：资金安全与数据合规

**规范要求**：
- **数值精度与舍入**：金额字段使用 `BigDecimal` 或 `Long`（以最小单位存储，如分），明确舍入规则（四舍五入/截断），避免累计误差。
- **幂等与防重复**：所有写操作必须有唯一约束（如业务单号+日期+操作类型），防止重复记账、重复出款。
- **对账差异处理**：如果下游需要对账，需设计差异自动/人工处理流程（挂账、调账、冲正）。
- **审计日志**：所有资金操作记录不可篡改的审计日志（操作人、时间、前后值、原因），保存期限符合监管要求（通常≥5年）。
- **敏感信息保护**：银行卡号、身份证等在日志/文件/接口响应中脱敏（如掩码），存储加密。

**提供方式**：
```
金额类型：[BigDecimal / Long(分)]
舍入规则：[四舍五入 / 截断 / 银行家舍入]
唯一约束：[业务单号+日期+操作类型]
对账差异处理：[自动挂账 / 人工调账 / 冲正] [N/A]
审计日志保存期：[如5年]
敏感信息脱敏：[掩码规则，如卡号保留前4后4]
```

## 第三部分：技术架构

### 11. 复用与新建决策

**规范要求**：
- 明确参考现有哪条链路
- 明确复用哪些组件（持久化、监控、日志等）
- 明确新建哪些组件
- 明确是否允许修改现有代码

**为什么重要**：新建独立链路可降低回归风险但增加维护成本；复用现有链路可快速交付但可能受限于现有设计。需基于数据来源、格式、交互方式的差异程度决策。

**提供方式**：
```
参考链路：[现有类名/链路名] [N/A]
复用部分：[如"复用持久化、监控、日志"]
新建部分：[如"新建独立Service"]
允许修改现有代码：[是/否，范围]
```

### 12. 输出形态与技术选型

**规范要求**：
- 明确输出/交付形态（文件/接口响应/消息/DB记录等）
- 明确数据量级，由数据量驱动技术选型
- 模板/配置方式（DB配置/硬编码/配置文件）
- 大数据量场景应避免重量级模板引擎，选择高性能处理方式

**为什么重要**：模板引擎对百万级数据渲染有额外开销，简单格式直接构建更高效。技术选型应由数据量驱动，不是由偏好驱动。

**提供方式**：
```
输出形态：[文件(CSV/Excel/JSON/文本) / 接口响应 / 消息 / DB记录]
数据量级：[百条 / 万条 / 百万条]
处理方式：[模板引擎 / 直接构建 / POI / ORM / 由我评估]
模板/配置方式：[DB / 硬编码 / 配置文件] [N/A]
```

### 13. 并发模型与线程池设计

**规范要求**：
- 明确默认模式（串行/并行）
- 并行时明确：是否需要独立线程池、线程池参数、饱和策略
- 明确是否有非线程安全组件需要同步
- 明确是否需要配置化切换（热更新）

**为什么重要**：串行简单可靠，适合数据量小或时间充裕的场景。并行需要考虑线程池管理、资源隔离、非线程安全组件同步。应基于实际数据量决策，避免过度工程。

**提供方式**：
```
默认模式：[串行 / 并行]
并发度：[如3-5] [N/A]
独立线程池：[是/否，理由] [N/A]
线程池参数：[核心/最大/队列/存活时间/饱和策略] [N/A]
非线程安全组件：[如 SFTP Channel、SimpleDateFormat 需 synchronized] [N/A]
热更新：[是/否，配置项名] [N/A]
```

### 14. 事务边界设计

**规范要求**：
- 明确哪些操作必须在同一事务内
- 明确事务传播行为（特别是嵌套调用时）
- 明确事务超时设置
- 不可回滚操作（文件上传、外部API调用）与 DB 操作不在同一事务

**为什么重要**：不可回滚操作 + DB 操作如果放在同一事务，外部操作失败时 DB 记录回滚，但外部副作用已产生，造成不一致。应拆分为独立步骤，用状态字段跟踪。

**提供方式**：
```
事务范围：[哪些操作在同一事务]
不可回滚操作处理：[分离/合一，状态跟踪方式] [N/A]
事务超时：[如30秒]
```

### 15. 代码修改边界

**规范要求**：
- 明确是否允许修改现有类
- 明确只允许新增的范围
- 明确允许改造的范围（如工具类新增方法、枚举新增值）
- 明确是否需要保持向后兼容

**为什么重要**：修改现有代码有回归风险，新增代码可降低风险。但有时必须改造（如工具类缺少需要的方法）。明确边界可避免在不确定时过度保守或过度修改。

**提供方式**：
```
允许修改现有类：[是/否，哪些]
只允许新增：[范围]
允许改造：[范围，如工具类新增方法]
向后兼容：[是/否]
```

## 第四部分：可靠性设计

### 16. 降级与开关设计

**规范要求**：
- 设计手动开关（总开关 + 黑名单/白名单），支持配置中心热更新
- 配置为空时的行为要明确（抛异常不降级 / 跳过 / 使用默认值）
- 降级层数应与任务关键性匹配，避免过度工程

**为什么重要**：总开关用于灰度发布和紧急关闭，成本极低收益明确。黑名单用于跳过异常数据维度。但过多降级层（如白名单+6种兜底+6种资源控制）属于过度工程，增加维护成本。

**提供方式**：
```
总开关：[配置项名，默认值]
黑名单/白名单：[配置项名] [N/A]
配置为空时行为：[抛异常 / 跳过 / 默认值]
降级策略偏好：[最简可靠 / 多层防护 / 由我评估]
```

### 17. 重试策略

**规范要求**：
- 区分需要重试和不需要重试的操作
- 网络类操作（HTTP/RPC/SFTP等）重试，DB 查询一般不重试
- 重试次数建议3次，间隔递增（如5s/10s/30s）
- 重试前重建连接（连接可能已失效）

**为什么重要**：网络操作可能因瞬时故障失败，重试可恢复。DB 查询失败通常是查询条件或数据问题，重试无意义。重试前不重建连接会用已失效的连接重试，必然再次失败。

**提供方式**：
```
需要重试的操作：[如HTTP调用 / SFTP上传 / RPC调用] [N/A]
重试次数：[如3次]
重试间隔：[如5s/10s/30s递增]
重试前重建连接：[是/否]
不需要重试的操作：[如DB查询]
```

### 18. 补偿与恢复机制

**规范要求**：
- 明确失败任务是否需要自动补偿
- 补偿触发时机（每次正常执行后 / 独立补偿任务）
- 补偿范围（查前 N 天的失败记录）
- 补偿去重规则（避免重复补偿同一失败）
- 补偿仍失败的处理（告警 + 人工介入）

**为什么重要**：自动补偿可在不影响正常流程的情况下恢复失败任务。去重规则避免重复补偿。用失败记录自己的业务日期重跑（而非用当前日期），确保数据一致性。

**提供方式**：
```
补偿方式：[自动(正常执行后) / 独立任务 / 手动 / 不需要] [N/A]
补偿范围：[如"查前7天失败记录"]
去重规则：[如"按维度+日期去重"]
补偿失败处理：[告警 + 人工介入]
```

### 19. 异常隔离策略

**规范要求**：
- 明确异常隔离粒度（单维度/单批次/单条）
- 隔离层 catch 后的动作：记录失败状态 + 记录监控 + 继续下一个
- 最外层 catch 不向上抛出（避免影响其他维度/日期）
- 异常分类：可重试异常 vs 不可重试异常

**为什么重要**：单维度失败不应影响其他维度。最外层不抛出避免调度框架/入口层将整个任务标记为失败。区分可重试/不可重试异常可避免对不可恢复的错误无意义重试。

**提供方式**：
```
隔离粒度：[如"单维度try-catch" / "单条"]
catch后动作：[记录失败状态 + 监控 + continue]
最外层策略：[catch不抛]
异常分类：[可重试(网络/超时) vs 不可重试(数据问题/业务校验)]
```

### 20. 边界数据处理

**规范要求**：
- 空数据：生成空结果 / 跳过 / 返回默认结构
- 超大数据：是否分批输出 / 单批次上限
- 脏数据：字段缺失/格式异常时跳过该条还是中断处理
- 跨边界数据：日期边界附近的数据归属哪天

**为什么重要**：空数据处理取决于对端是否检查结果存在。脏数据处理方式影响整体流程的健壮性——跳过单条比中断整个维度更合理。日期边界取决于业务定义（按会计日期还是按交易日期）。

**提供方式**：
```
空数据：[空结果 / 跳过 / 默认结构]
超大数据：[分批输出 / 单批次，上限] [N/A]
脏数据：[跳过单条+记录告警 / 中断处理]
日期边界：[按会计日期 / 按交易日期] [N/A]
```

### 补充：高可用与容灾

**规范要求**：
- **多活/灾备**：如果依赖的数据库或外部服务是多活架构，需明确是否支持异地切换、切换后任务能否自动恢复。
- **优雅关闭**：任务被 kill 时保证正在处理的批次不丢失（注册 ShutdownHook、等待当前批次完成）。
- **限流与熔断**：对外部依赖增加限流（令牌桶）和熔断（断路器），防止雪崩。
- **容量规划**：根据历史数据增长趋势，定期评估线程池、连接池、磁盘空间是否需要扩容。

**提供方式**：
```
多活/灾备：[是/否，切换策略] [N/A]
优雅关闭：[ShutdownHook / 等待当前批次] [N/A]
限流：[令牌桶，速率] [N/A]
熔断：[断路器，阈值] [N/A]
容量规划：[评估周期，扩容阈值] [N/A]
```

**提供方式**：

## 附录：通用设计原则

> 从实战项目提炼，每条原则附带典型反面案例。

| # | 原则 | 反面案例 |
|---|------|---------|
| 1 | **数据量驱动架构决策**：先调研实际数据量，再决定是否需要并发/流式/降级。不过度工程，也不工程不足。 | 按最坏场景设计独立线程池+5层降级，实际27维度串行15分钟完成，全部多余 |
| 2 | **分片表查询提前确认兼容性**：分片表广播查询可能触发中间件 ResultSet 兼容性问题，提前设计原生 SQL + 索引映射。 | 编码后才发现广播查询不支持按列标签取值，ORM 实体映射失败，排查耗时最长 |
| 3 | **状态常量数值重叠检查**：不同业务类型共用同一状态字段但数值含义不同时，必须按类型分别映射。 | 支付 status=1=已成功，退款 status=1=已退款，同一映射函数导致退款订单状态错误 |
| 4 | **对端格式规范需求阶段获取**：样例数据+对端代码/接口定义应在需求阶段完整获取，减少编码期 TODO。 | 17处字段映射 TODO 阻塞上线，根因是对端代码未在需求阶段提供 |
| 5 | **技术选型由数据量驱动**：百万级数据用直接构建+缓冲处理，千级可用模板引擎，不要由偏好驱动。 | 百万行用模板引擎渲染开销大，改用 StringBuilder+BufferedWriter 后性能达标 |
| 6 | **游标分页避免深分页**：大数据量用游标分页（如 order_id 倒序），避免 offset 深分页性能退化。 | offset 分页到第100页时 OFFSET 990000 查询急剧变慢 |
| 7 | **外部依赖需求阶段确认**：连接信息来源和到位时间在需求阶段确认，避免编码完成后无法部署。 | SFTP 连接信息一直未到位，编码完成后阻塞部署 |
| 8 | **独立监控枚举**：新增功能用独立监控事件枚举，与现有枚举区分，成本极低收益明确。 | 共用现有枚举导致无法区分哪个功能失败 |
| 9 | **补偿用失败记录自己的业务日期**：补偿时用失败记录的业务日期重跑，而非用当前日期，确保数据一致性。 | 用当前日期重跑导致查到错误日期的数据 |
| 10 | **降级层数与关键性匹配**：总开关+异常隔离+补偿三层足够，避免白名单+多种兜底+多种资源控制的过度工程。 | 5层降级+6种兜底+6种资源控制，实际全部用不到，增加维护成本 |
| 11 | **不可回滚操作与DB操作分离**：文件上传/外部API调用等不可回滚操作与DB操作不在同一事务，用状态字段跟踪各步骤。 | 文件上传失败时 DB 记录回滚，但文件已生成在本地，造成不一致 |
| 12 | **配置为空时关键配置抛异常**：关键配置（如商户列表）为空应抛异常不降级，避免静默跳过。 | 商户列表为空时静默跳过，实际是配置错误但无任何告警 |
